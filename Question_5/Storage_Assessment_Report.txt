4. INODE UTILISATION
----------------------------------------------------------------
Command used:
    $ df -i

Output (key row):
    Filesystem   Inodes    IUsed    IFree  IUse% Mounted on
    /dev/vda   16777216   193999 16583217    2%  /

Observations:
  * Only 2% of inodes are used, so there is no risk of running
    out of inodes.
  * A filesystem can fill up in TWO ways: out of blocks (space)
    OR out of inodes (too many tiny files). Both must be checked;
    df -h alone is not enough.

----------------------------------------------------------------
5. STORAGE HEALTH OBSERVATIONS
----------------------------------------------------------------
  * Space usage at 47% and inode usage at 2% - both healthy.
  * No removable or failing devices detected.
  * Read-only squashfs mounts reduce accidental-corruption risk.
  * Available space on '/' (~10 G) is the main item to watch for
    a growing application server with logs and data.

----------------------------------------------------------------
6. RECOMMENDATIONS FOR IMPROVING STORAGE UTILISATION
----------------------------------------------------------------
  1. Set up monitoring/alerts on df output (e.g. alert at 80%).
  2. Put application data and logs on a SEPARATE partition or
     mounted volume so a log flood cannot fill '/'.
  3. Implement log rotation (logrotate) to cap log growth.
  4. Schedule cleanup of temp files and old artefacts (cron).
  5. Consider LVM so the volume can be extended online if the
     application grows.
  6. Track inode usage on filesystems that store many small
     files, not just block space.
